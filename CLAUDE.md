# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project: SpectrumAnalyzer

An AudioMuse-AI plugin for audio spectrum analysis that detects fake lossless files and transcodes via frequency-cutoff heuristics. Results include spectrograms (base64 PNG) and verdicts (CLEAN, CONSISTENT_LOSSY, LOWPASSED, TRANSCODED_LOSSY, FAKE_SUSPECT).

## Architecture

### Three-module design

- **`__init__.py`** — Flask blueprint for the web UI (album overview, per-track detail, settings). Handles database migration, plugin registration, and routes for scan/rescan/settings actions. Verdicts are color-coded; spectrograms are rendered inline.

- **`jobs.py`** — Background job logic. Three scan modes:
  - `changed`: skip tracks whose metadata fingerprint (path, size, bitrate, etc.) hasn't changed
  - `verify`: download every file, re-analyze only if audio MD5 differs (catches in-place edits invisible to metadata)
  - `force`: re-download and re-analyze everything
  - `scan_library_job()` is a parent orchestrator (must run on the **high** queue): it fans out one `scan_album_job()` per album onto the default queue with `parent_task_id`, throttles in-flight children to `config.MAX_QUEUED_ANALYSIS_JOBS`, and tracks done/remaining by polling the children's `task_status` rows (statuses + counters aggregated from their details JSON). In `changed` mode it settles all-unchanged albums itself without spawning a child. Cancellation: it checks its own row for REVOKED each poll; children check the parent's status at start. This mirrors core's `run_analysis_task`/`analyze_album_task` pattern.
  - `scan_album_job()` also runs standalone (no parent) for the album Re-analyze button.
  - `on_song_analyzed()` hook piggybacks on core analysis when the audio is already on disk (no extra download).

- **`dsp.py`** — Audio DSP (librosa + numpy) and analysis pipeline:
  - `_load_segment()` extracts one fixed-offset segment (offset 30s, falling back to 0s for short files) — **not** the middle of the track despite the old docstring; now only the last-resort fallback for `_load_windows()` (0.6.0, distributed segment sampling — see below).
  - **Distributed segment sampling (0.6.0)**: `_load_windows()` replaces the old single fixed-offset segment with several windows spread through the track, each scored independently, so one bad-luck sample can't miss a low-bandwidth passage elsewhere in the file. Two tiers, keyed off `_probe_container()`'s tier-2 result (`detected_fmt is not None` = SoundFile can open it): `_read_window_soundfile()` seek-and-read for FLAC/WAV/AIFF/MP3/OGG (`_NATIVE_WINDOW_POSITIONS`, 5 windows at ~10/30/50/70/85%, no subprocess cost) vs. `_read_window_ffmpeg()` (`-ss`/`-t` input-seeking to raw `f32le`, one ffmpeg spawn per window) for everything else — m4a/mp4/opus/wma/mpc — (`_FFMPEG_WINDOW_POSITIONS`, 3 windows at ~20/50/80%, the sample rate/duration for which comes from `_probe_ffprobe()`). `segment_seconds` keeps its existing meaning as the total decode budget, now `/= len(positions)` per window. Each window gets its own `_find_cutoff()`; windows within `_WINDOW_NEAR_SILENT_REF_DB = -85 dB` of true silence (not merely quiet content) are skipped via `_is_near_silent()`. The **minimum** cutoff among the surviving windows — not an average — becomes the primary window that drives `_edge_sharpness()`/`_shelf_level()`/`_verdict()`/the rendered spectrogram/`seg_offset`/`seg_seconds`, exactly as the old single window did; the other windows are reported alongside it in `details.windows.samples` (offset/seconds/cutoff_hz/silent) with `details.windows.agree` (spread ≤ `_WINDOW_AGREEMENT_TOLERANCE_HZ = 1500 Hz`, `null` when fewer than 2 valid windows) as an additive, UI-free signal ("disagreement" = a reason to deep-scan); `details.windows` itself is `null` in deep mode, where no windowing happens. `_load_windows()` falls back to the old single `_load_segment()` window when duration can't be determined, the file is too short to distribute, or every window in a tier fails to decode (e.g. no ffmpeg on PATH). Deep mode (`_deep_spectrum()`) already scans the whole file and is unaffected. `ANALYSIS_REV` bumped to 4.
  - `_spectrum()` computes STFT, reduces to per-frequency 95th-percentile profile (robust against silence/clicks).
  - `_find_cutoff()` (0.6.0, robust cutoff detection) locates the highest frequency within a drop threshold (default: 40 dB below 1–8 kHz reference) over a sustained band: at least 250 Hz wide with 75% of bins above threshold (gaps up to 120 Hz bridged first — real spectra have harmonic notches, an unbroken run is too strict), profile lightly smoothed (100 Hz) first. All these widths are in hertz, converted to bins from the file's actual bin resolution (`sr/N_FFT`) so behavior doesn't silently vary with sample rate. Isolated runs narrower than 80 Hz are excluded before gap-closing (so a lone pilot tone/spike can't get bridged into a real band and drag the cutoff up) and surfaced as the nullable `details.evidence.narrow_high_frequency_tone_present` flag when one sits ≥ 8 kHz. `ANALYSIS_REV` bumped to 2.
  - `_edge_sharpness()` measures rolloff steepness in dB/kHz (encoder low-pass filters are near-vertical; natural rolloff is gradual).
  - `_verdict()` decision logic: cutoff ≥ min(0.93 × Nyquist, 20.5 kHz) → CLEAN (genuine masters roll off at 20–21 kHz; near-Nyquist edges are meaningless), unless the container is hi-res (> 48 kHz sample rate) and the cutoff aligns with a standard lower rate's Nyquist (`_resample_match`, asymmetric −3%/+7% window: resampler leakage pushes detected cutoffs up) or sits below 23 kHz — then UPSAMPLED **only with corroboration** (silent shelf, aliasing images via `_best_alias_match` mirrored around candidate source Nyquists, or sharp edge); without corroboration it's a CLEAN "dark master?" note. Lossless + sharp edge + silent shelf (≤ −68 dB vs. reference — calibrated to real 16-bit re-encodes whose dither refills encoder silence at −73…−106; genuine noise sits ~−53) → FAKE_SUSPECT (this catches LAME 320→FLAC; cutoffs ≥ 20.5 kHz like AAC 256 still pass); sharp edge but audible noise above cutoff → LOWPASSED; gradual rolloff → LOWPASSED; lossy with low cutoff relative to declared bitrate → TRANSCODED_LOSSY **only for a codec in `_CALIBRATED_TRANSCODE_CODECS`** (currently `{'mp3'}` — the only codec `_expected_cutoff_for_bitrate`'s table is calibrated against; ffmpeg's native AAC/WMA legitimately cut lower than MP3 at the same declared bitrate, so uncalibrated codecs get a "no calibrated mapping" note instead of a verdict change); the MP3 gate itself never fully trusts the evidence either (`_TRANSCODE_GATE_CONFIDENCE_PENALTY = 0.10`, floored at 0.5, plus a note) since an intentionally low-passed first-generation MP3 is a legitimate alternative explanation — see `_resolve_lossy_codec()` for how the codec name reaches the gate (probed value, or suffix-trust for mp3 only, so the shipped 128-as-320 fixture keeps firing even without ffprobe); otherwise CONSISTENT_LOSSY. `ANALYSIS_REV` bumped to 3 (codec-aware transcode gating changes verdicts for non-MP3 lossy codecs and confidence for MP3).
  - `_bit_depth_probe()` reads PCM via soundfile as int32 (left-justified) and computes effective bit depth from trailing zero bits: a >16-bit container with ≤16 effective bits is fake 24-bit → UPSCALED verdict (when spectrum is otherwise CLEAN) or a note. Stored as `container_bits`/`effective_bits` columns (meanings unchanged since 0.5.0).
  - **Bit-depth histogram (0.6.0)**: `_bit_depth_probe(path, full=deep)` builds a 33-bucket histogram of per-sample effective bit depth (`_bit_depth_window_stats`, shared by both modes and independently unit-testable) alongside the pre-existing exact minimum — the exact test itself (every tested nonzero sample padded) is unchanged and stays the *only* UPSCALED trigger. Segment mode reads the same ~30 s middle window as before (`details.bit_depth.coverage = 'sampled'`, confidence **0.9** when exact); deep mode instead chunks through the whole file bounded at `DEEP_MAX_SECONDS` like `_deep_spectrum()` (`coverage: 'full'`, confidence **0.95** when exact, or `'capped'` — still 0.95 — if the time budget was hit first). When the exact test fails but the histogram's mode is still ≤ 16 bits, a note reports `predominant_bit_depth` + `lower_bit_activity_fraction` (the fraction of samples with real content below 16 bits) as purely informational statistical evidence — dither, gain changes, SRC, or edits can all leave a few genuinely deeper samples without the file being any less of an upscale; this never changes the verdict. `details.bit_depth` gains these two fields plus `coverage` alongside the existing `container_bits`/`effective_bits`. `ANALYSIS_REV` bumped to 5 — the previously hardcoded 0.95 confidence for an exact UPSCALED hit is now 0.9 for a sampled-window confirmation and 0.95 only for a deep-mode full-file one, a real confidence correction on existing rows, not just a `details`-shape change.
  - **Codec probe (0.6.0), all 3 tiers**: `_sniff_dsd()` recognizes DSF/DFF by magic bytes (content, not extension) *before* any decode is attempted — the audioread/ffmpeg fallback would otherwise silently decode DSD noise shaping as full-bandwidth PCM and read it as CLEAN; a match short-circuits `analyze_file` straight to an `INCONCLUSIVE`/`unsupported` result via `_unsupported_format_result()`, `deep_eligible=False`. `_probe_container()` (tier 2) calls `soundfile.info()` to identify the actual container for formats libsndfile reads natively (FLAC/WAV/AIFF/MP3/OGG — verified libsndfile 1.2.2 reports these distinctly), also reading `.subtype` for the lossy formats (e.g. `MPEG_LAYER_III` → `mp3`) at no extra cost. `_probe_ffprobe()` (tier 3; renamed/extended from `_probe_codec_ffprobe` in 0.6.0) shells out to `ffprobe` once for codec **and** sample_rate **and** duration together — the codec lookup runs when a lossy suffix reaches `analyze_file` with no codec resolved yet (m4a/mp4/ogg-opus/wma/mpc — libsndfile can't open these), and the same call is reused (memoized) for the ffmpeg-tier windowing above when the container isn't SoundFile-seekable. This is what fixes ALAC-in-.m4a: an `m4a` suffix (always suffix-classified lossy otherwise) that ffprobe identifies as `alac` flips `is_lossless` to `True`. Any failure (missing binary, timeout, bad output) returns all-`None` — "unknown codec" — never a guess. Feeds an `is_lossless` override into `_verdict()` (default `None` = fall back to suffix-based `LOSSLESS_SUFFIXES`/`LOSSY_SUFFIXES` membership, unchanged). A probed/suffix disagreement sets `details.delivery.codec_mismatch`; the resolved codec itself (when any tier confirms one) is `details.delivery.codec`.
  - **Progressive integrity (0.6.0)**: every result's `details.integrity = {status, coverage}`. `status` ∈ `unsupported` (DSD)/`decode_failed`/`sampled_decode_ok`/`full_decode_ok`; `coverage` ∈ `null`/`sampled`/`capped`/`partial`/`full`. Segment mode is always `sampled_decode_ok`+`sampled`. Deep mode: `_deep_spectrum()` reports `capped` (`total >= DEEP_MAX_SECONDS`, purely arithmetic) and `chunk_error` (the loop's `except Exception: break` fired — a real decode error, structurally distinct from the two natural-end-of-file break sites) — `full_decode_ok`+`full` only when neither fired; `chunk_error` → `sampled_decode_ok`+`partial`; `capped` → `sampled_decode_ok`+`capped`. Deliberately not attempted: distinguishing a genuine mid-file decode error from a backend that raises at true EOF instead of returning a short read — that's what the later "Minimum adversarial fixtures" item validates.
  - **Structured `details` namespacing (0.6.0)**: every fact in `details` has exactly one namespaced owner — decode state in `integrity.*`, codec mismatch/resolved codec in `delivery.*`, per-window data in `windows.*`, bit-depth columns in `bit_depth.*`, and the `evidence` namespace for flags with no dedicated section (`edge_machine_like`, `shelf_digitally_silent`, `alias_image_detected`, `narrow_high_frequency_tone_present`). New/future flags should extend the matching namespace rather than adding a flat top-level key. Evidence flags are nullable: `null` = not evaluated (e.g. `alias_image_detected` stays `null` outside `_verdict()`'s UPSAMPLED-candidate branch, where the alias-image correlation never runs), `false` = tested and absent — `_edge_sharpness()` returns `None` (not a misleading `0.0`) when there isn't enough band above the cutoff to measure, mirroring `_shelf_level()`'s existing `None`-for-unmeasurable contract, so `edge_machine_like` can distinguish "measured, not sharp" from "not measured" while the verdict-driving `sharp` boolean (`edge_db_khz is not None and edge_db_khz >= 25.0`) stays behaviorally unchanged. `details.analysis_rev` mirrors the top-level column for raw-dump auditability. Purely a `details`-shape change — no verdict/confidence value changed, `ANALYSIS_REV` not bumped for it.
  - `_deep_spectrum()` (deep=True on `analyze_file`, from the per-track Deep analyze button): chunked whole-file scan (120 s chunks, 30 min cap) producing a global max-hold profile, a 1-column/second spectrogram, and a per-second spectral-edge series. Measured calibration: encoder wall ±198 Hz, content-varying upsample ±1035 Hz, genuine dark master ±7236 Hz → edge variance ≥ 2000 Hz downgrades UPSAMPLED/FAKE_SUSPECT → LOWPASSED "likely genuine dark master"; ≤ 300 Hz (constant wall) raises confidence; a pinned edge (≥50% of seconds within 4% of a standard Nyquist, ≤5% above it) confirms a resampler wall regardless of variance. Quiet seconds (window ref < chunk ref − 25 dB) are skipped.
  - `verified` column (0.5.0): manual per-track flag toggled from the album page; excluded from all suspect counts/filters in `__init__.py` (`bad_expr`), preserved across re-analysis (not in the upsert column list).
  - `deep_pending` column (0.5.1): set by the `deep_rescan` route at queue time, cleared in `analyze_track_job`'s finally block (survives job failure). Shown as a "deep scan queued" badge on the song card and aggregated into per-album overview tags (deep scan ×N / verified ×N).
  - `_render_spectrogram()` matplotlib-based PNG (spek-style color map, red line at detected cutoff), base64 encoded, tunable in settings.

### Database

Results table with foreign key to core's `score(item_id)`, declared `ON UPDATE CASCADE ON DELETE CASCADE`. The UPDATE cascade is load-bearing: AudioMuse-AI 3.0 relabels `score.item_id` in place (native media-server ids → canonical `fp_...` ids) and only detaches its own FKs around that UPDATE — a plain FK blocks the whole core migration, the cascade migrates our rows losslessly. Fingerprints and MD5s stored to avoid redundant downloads/analysis.

**Analysis-revision stamp (0.4.0)**: `analysis_rev` column stores `dsp.analysis_rev(drop_db, segment_seconds)` (= hand-bumped `dsp.ANALYSIS_REV` folded with the verdict-relevant settings; rendering settings excluded). Every skip path requires a rev match in addition to its fingerprint/MD5 match — changed-mode fingerprint skip, verify/changed MD5 skip, the `on_song_analyzed` hook skip, and `scan_library_job`'s settle-without-child path. Bump `ANALYSIS_REV` whenever verdict logic changes; NULL rows (pre-0.4.0) re-analyze on the next non-force scan.

**Verdict/substatus contract (0.4.0)**: `INCONCLUSIVE` is the one new primary verdict for files analysis couldn't meaningfully run on — currently just decode failure (`dsp.analyze_file` catches `_load_segment`/`_deep_spectrum`'s `ValueError` and returns a normal-shaped result instead of raising, so decode failures flow through `_upsert` like any other verdict instead of vanishing silently from `scan_album_job`'s error counter or `on_song_analyzed`'s swallowed exception). Excluded from `SUSPECT_VERDICTS`/`bad_expr` but never invisible — the overview page carries a dedicated `n_inconclusive` count alongside suspect/lowpassed/verified, and it gets a filter checkbox for free via `VERDICT_STYLE`. Deep-scan eligibility is a separate `deep_eligible` column (`BOOLEAN NOT NULL DEFAULT TRUE`), not derived from the primary verdict — decode failure sets it `False` (retrying won't help); both deep-scan gates (`deep_rescan` route, `_queue_deep_non_clean`) and the per-track "Deep analyze" button check it. Unsupported-codec and DSD-without-ffmpeg substatuses are deliberately deferred to the container/codec-probe item, which is what will actually detect them.

**ID model (0.3.0)**: `item_id` = whatever `score` is keyed by (native id on 2.x, canonical fp id on 3.x — same code handles both); `provider_track_id`/`server_id` columns keep the native id for `download_track`/rescans.

**Task-row ids (0.3.0)**: task_status rows for top-level jobs MUST be keyed by the RQ job id — the 3.x janitor (`rq_janitor` → `reap_orphaned_tasks`, 120 s grace) fails any top-level non-terminal row whose id `Job.fetch` can't find, so a route-invented uuid row is reaped mid-scan. Routes therefore never pre-create rows (also because `run_plugin_task` bare-SUCCESS-overwrites, details wiped, any row keyed by the job id that exists before the task runs); jobs derive their id via `jobs._rq_task_id()` and create their own row. Parent-launched children keep uuid ids — they're janitor-immune via `parent_task_id` and reconciled by `_rq_job_lost`.

**Multi-server (0.3.0)**: the core's server binding (`plugin.api.use_server`) is a contextvar that does NOT cross RQ job boundaries, so every job binds its own server via `jobs._bind_server` (nullcontext on 2.x / server None = default). UI scans pass `all_servers=True` → `scan_library_job` plans `(server_id, album)` pairs over `_list_server_ids()` and launches each `scan_album_job` with its `server_id`; standalone album re-analyze recovers it from stored rows (`_album_server`); `_analyze_track` binds `row_server` so re-downloads hit the right catalogue. Cron entry (`scan_changed_task`) stays single-run — a schedule's server scope makes the core run it once per server, already bound (`_active_server_id()` short-circuits the fan-out). Tracks shared between servers collapse onto one canonical id (no duplicate rows); their per-server path/size metadata can flip the fingerprint between alternating scans, costing a re-download that ends `unchanged` via the MD5 check. Translation happens at the edges via `jobs._native_to_fp`/`_fp_to_native` (wrapping `tasks.mediaserver.registry.reverse_translate_ids`/`translate_ids`, identity/no-op fallback on 2.x). One batched translation per album, mirroring core's `attach_catalog_item_ids`. The v3 `on_song_analyzed` payload's `item_id` is the NATIVE id — the canonical id is `media_item['_catalog_item_id']`. `migrate()._migrate_v3_ids` = columns → defensive re-key via `track_server_map` → FK repair (NOT VALID + best-effort VALIDATE) → provider backfill; all idempotent, tested against live Postgres in `scratchpad` (four upgrade scenarios).

### Plugin integration points

- Settings page for segment length, cutoff threshold, spectrogram dimensions, and optional on_song_analyzed hook toggle.
- Optional cron task (`plugin.spectrum_analyzer.scan_changed`, seeded disabled, Sunday 04:00).
- Manual re-analyze button per track (enqueued to the `high` queue).

## Development workflow

- **Run tests**: `python3 -m unittest discover tests -v` from the repo root (needs `numpy soundfile matplotlib` via pip; librosa is stubbed by `tests/librosa_stub.py`). `tests/fixtures/` holds committed 45 s ground-truth files (genuine masters + every fake type built from a confirmed-genuine DSD source). Always run after touching `dsp.py`.
- **Ad-hoc analysis**: `python3 tests/run_verdicts.py "path::suffix::kbps::EXPECTED::seg|deep"`.
- **Build**: `./build.sh [version|local]` — `local` replaces `dist-local/spectrum_analyzer.zip` for the fast dev loop (`dist-local/` carries its own fixed manifest.json/plugin.json catalog, always pointing at that unversioned zip; never touched by the script). A version (or no argument = newest zip in `dist/`, falling back to `plugin.json`) builds `dist/spectrum_analyzer-<version>.zip` and rewrites the matching `plugin.json` entry's `sourceUrl`, warning if the version has no entry yet. Old versioned zips are kept so published entries stay downloadable.
- **Release** (CI/CD, `.github/workflows/build.yml`): releases happen only when a **PR is merged into main**. The PR must change the code *and* add a new top entry (version + changelog) to `plugins/SpectrumAnalyzer/plugin.json` (full history in `CHANGELOG.md`). Tests run on every PR update and again on the merge. Before building, CI **fails** if (a) any published zip is missing or its md5 no longer matches its plugin.json checksum, or (b) plugin code changed but the newest version is already built (no bump). Otherwise it builds `dist/spectrum_analyzer-<version>.zip` and commits it plus the entry's `sourceUrl`/md5 `checksum` back (`[skip ci]`). Direct pushes to main do not release — protect main so everything lands via PRs. **Published versions are immutable** (installed copies re-verify by checksum, e.g. after a plugins-volume wipe); build.sh enforces this locally too (`FORCE=1` only for never-published versions). Per core docs: blueprint name must equal the plugin id, the zip must not contain plugin.json, `on_install` must be re-runnable, `requirements` carries pip version pins. Community-catalog listing = a one-entry PR to NeptuneHub/AudioMuse-AI-plugins' `manifest.json` pointing `pluginUrl` at this repo's raw `plugins/SpectrumAnalyzer/plugin.json` (the externally-hosted model used by the Lumae plugins) — this repo stays the single source of truth for code, releases and checksums.
- **Regenerate fixtures**: `./tests/generate_fixtures.sh <genuine-hires-file> tests/fixtures [ffmpeg] [start] [dur]` (committed set: start=60, dur=45). Calibration constants measured from these fixtures are documented in `tests/README.md` — update both when retuning thresholds.
- The web/jobs layers (`__init__.py`, `jobs.py`) still require a live AudioMuse instance to test.
- **Dependencies**: librosa, numpy (included in AudioMuse-AI), matplotlib (declared in `plugin.json`, auto-installed).
- **Change detection strategy** is a two-layer optimization: (1) metadata fingerprint for cheap skips, (2) audio MD5 to catch byte-level changes. This avoids downloading unchanged files during incremental scans.
- **Spectrogram caching**: base64-encoded PNG stored in Postgres. Database size ~100–300 KB per spectrogram (default 800×280 px); 10k songs ≈ 1–3 GB.
